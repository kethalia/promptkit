# Document Function

Generate comprehensive documentation for a function in the appropriate format for its language.

---

## Context

Gather before starting:
- Function source code and file location
- Language and existing documentation style in codebase
- Type definitions or interfaces used
- Call sites showing how function is used
- Related functions in same module
- Test files covering the function

## Instructions

1. **Analyze function signature**
   - Parameters: names, types, default values, optionality
   - Return type and possible return values
   - Generic/template parameters if present
   - Overloads if applicable

2. **Understand implementation**
   - Core logic and algorithm
   - Branches and edge cases handled
   - Error conditions and thrown exceptions
   - Side effects (mutations, I/O, state changes)

3. **Find usage patterns**
   - Search for call sites in codebase
   - Review test cases for expected behavior
   - Note common parameter combinations
   - Identify edge cases from tests

4. **Determine documentation format**
   - JavaScript/TypeScript: JSDoc or TSDoc
   - Python: Google, NumPy, or Sphinx docstring style
   - Rust: Doc comments with examples
   - Go: Godoc conventions
   - Match existing style in codebase

5. **Write documentation**
   - Brief description (first line)
   - Detailed explanation if needed
   - Document each parameter
   - Document return value
   - Document errors/exceptions
   - Add usage examples
   - Note related functions with @see or equivalent

6. **Validate documentation**
   - Ensure types match actual signature
   - Verify examples would compile/run
   - Check for outdated information

## Output Format

### TypeScript/JavaScript (TSDoc)
```typescript
/**
 * Brief description of what the function does.
 *
 * Longer description with details about behavior,
 * algorithm, or important notes.
 *
 * @param paramName - Description of parameter
 * @param options - Configuration options
 * @param options.key - Description of option
 * @returns Description of return value
 * @throws {ErrorType} When condition occurs
 *
 * @example
 * ```ts
 * const result = functionName('input', { key: 'value' });
 * console.log(result); // expected output
 * ```
 *
 * @see relatedFunction
 */
```

### Python (Google style)
```python
"""Brief description of what the function does.

Longer description with details about behavior,
algorithm, or important notes.

Args:
    param_name: Description of parameter.
    options: Configuration options.
        key: Description of option.

Returns:
    Description of return value with type info.

Raises:
    ErrorType: When condition occurs.

Example:
    >>> result = function_name('input', key='value')
    >>> print(result)
    expected output

See Also:
    related_function: Brief description.
"""
```

### Rust
```rust
/// Brief description of what the function does.
///
/// Longer description with details about behavior,
/// algorithm, or important notes.
///
/// # Arguments
///
/// * `param_name` - Description of parameter
/// * `options` - Configuration options
///
/// # Returns
///
/// Description of return value
///
/// # Errors
///
/// Returns `ErrorType` when condition occurs
///
/// # Examples
///
/// ```
/// let result = function_name("input", Options::default());
/// assert_eq!(result, expected);
/// ```
///
/// # Panics
///
/// Panics if condition occurs
```

### Go
```go
// FunctionName does brief description.
//
// Longer description with details about behavior.
// Parameter details can be included here.
//
// Example:
//
//	result := FunctionName("input", opts)
//	fmt.Println(result) // expected output
```

## Interactive Decisions

Ask the user about:
- **Documentation style**: Which docstring format to use if not clear from codebase?
- **Example complexity**: Simple or comprehensive with edge cases?
- **Internal details**: Document implementation details or just public contract?
- **Related functions**: Include cross-references to related code?
- **Deprecation**: Is function deprecated? What replaces it?
