# Review Documentation Sync

Review code documentation and verify it is synchronized with actual code implementation, ensuring docs reflect current functionality, APIs, parameters, and behavior.

---

## Context

Gather before starting:
- All documentation files (README, guides, API docs)
- Source code and implementation files
- Recent git history to understand changes
- Test files showing current usage
- Breaking changes or deprecations
- Migration guides if versions have changed

## Instructions

1. **Identify code changes and git history**
   - Review recent commits to understand what changed
   - Identify modified functions, classes, and modules
   - Note added or removed features
   - Find renamed parameters or changed signatures
   - Look for version bumps and releases

2. **Map documentation to code**
   - List all functions/classes documented
   - Find corresponding implementation in codebase
   - Identify undocumented new code
   - Note documented code that was removed
   - Track functions that moved to different modules

3. **Compare function signatures**
   - Verify parameter names match implementation
   - Check parameter types are accurate
   - Validate default values are correct
   - Ensure optional/required status matches
   - Check return types match actual returns
   - Verify thrown errors match implementation

4. **Validate behavior descriptions**
   - Ensure documented behavior matches implementation
   - Check edge cases are handled as documented
   - Verify error conditions match code
   - Validate performance characteristics if mentioned
   - Check side effects are accurately described

5. **Review code examples**
   - Run or validate examples against current API
   - Check imports use correct module paths
   - Verify function calls use current signatures
   - Ensure examples use non-deprecated methods
   - Validate configuration objects match schemas

6. **Check version information**
   - Verify version numbers match package manifest
   - Check changelog reflects actual changes
   - Validate "added in version X" claims
   - Ensure deprecation warnings are accurate
   - Check migration guides work with current version

7. **Identify documentation gaps**
   - Find new features without documentation
   - Note changed behavior not reflected in docs
   - Identify removed features still documented
   - Look for changed defaults not updated
   - Find new configuration options undocumented

- **Scope**: Review all code or specific modules/features?
- **Time range**: Review recent changes or entire codebase?
- **Git history depth**: How many commits back to analyze?
- **Auto-fix**: Should I update documentation automatically for minor issues?
- **Priority**: Focus on critical user-facing APIs or include internal docs?
- **Generate updates**: Just report issues or draft documentation updates?
- **Breaking changes**: Should I identify potential breaking changes from code evolution?
